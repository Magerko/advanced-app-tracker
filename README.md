# Advanced App Tracker

Десктопный трекер активности приложений на **PyQt6** с функциями самоконтроля.
Отслеживает, в каком приложении вы работаете, ведёт статистику по дням и
неделям, делит приложения на продуктивные/непродуктивные, позволяет ставить
лимиты времени и - при желании - **мешает себя закрыть** (пароль на выход +
сторож, перезапускающий приложение, если его «убить»).

## Возможности

- **Отслеживание активного окна** (Windows / macOS / Linux X11 и Wayland, с fallback по
  заголовку окна) и запись длительностей в SQLite.
- **Статистика**: суммарное время за сегодня и неделю, разбивка по
  продуктивности, история за произвольный период.
- **Графики**: круговая диаграмма распределения (matplotlib) и тренд дневного
  использования (pyqtgraph).
- **Лимиты** дневные/недельные с предупреждениями при приближении и
  опциональным завершением приложения при превышении.
- **Детекция простоя** - время бездействия не засчитывается.
- **Системный трей**: сворачивание вместо закрытия, запуск свёрнутым.
- **Самоконтроль (опционально)**:
  - пароль на выход из интерфейса;
  - **сторож (guardian)** - фоновый процесс, перезапускающий приложение, если
    его завершить (например, через Диспетчер задач). Корректный выход через меню
    сторож не трогает; есть защита от «шторма» перезапусков.
- **Автозапуск** при старте Windows.
- Тёмная тема.

## Установка

Нужен **Python 3.10+**.

```bash
pip install -r requirements.txt
```

Опциональные пакеты (`pynput`, `bcrypt`, `pygetwindow`) улучшают работу
(детекция простоя, надёжное хеширование пароля, кроссплатформенный fallback),
но без них приложение тоже запускается — соответствующие функции просто
отключаются.

## Запуск

```bash
python -m app_tracker
# или
python run.py
```

## Linux — поддержка Wayland (KDE Plasma)

На X11 всё работает из коробки. На **KDE Plasma Wayland** при первом запуске
приложение автоматически устанавливает небольшой KWin-скрипт
(`~/.local/share/kwin/scripts/activewindowtracker/`) и загружает его через DBus.
Скрипт подписывается на смену активного окна и пишет данные в journald —
приложение читает их в фоновом потоке.

После перезагрузки KWin подхватывает скрипт сам, ничего делать не нужно.

Требования для Wayland:
- `qdbus` (обычно уже есть в KDE, пакет `qt5-tools` или `qdbus-qt5`)
- `journalctl` (systemd)

## Структура проекта

```
app_tracker/
├── app.py                 # сборка QApplication и точка входа
├── config.py              # константы и ключи настроек
├── paths.py               # расположение БД/логов в каталоге пользователя
├── logging_setup.py       # настройка логирования (консоль + файл)
├── core/                  # движок: БД, детектор простоя, трекер
│   ├── database.py
│   ├── idle_detector.py
│   ├── tracker.py
│   └── productivity.py
├── platform_support/      # OS-специфика: активное окно, автозапуск, запуск
├── security/              # пароли и сторож (guardian + helper-процесс)
├── ui/                    # окно, диалоги, графики, тема
├── utils/                 # форматирование и время
└── resources/             # иконка
run.py                     # лаунчер (используется автозапуском и сторожем)
tools/make_icon.py         # генерация иконки
```

## Разработка

Зависимости для разработки (линтер и тесты):

```bash
pip install -e .[dev]
```

Проверка стиля и запуск тестов:

```bash
ruff check .
pytest
```

Тесты покрывают чистую логику без GUI: форматирование длительностей и парсинг
лимитов, границы недели, хеширование пароля (bcrypt и SHA-256-fallback),
классификацию продуктивности и слой работы с SQLite.

## Данные и логи

Хранятся в каталоге пользователя, а не рядом с кодом:

- Windows: `%LOCALAPPDATA%\AppTracker\`
- macOS: `~/Library/Application Support/AppTracker/`
- Linux: `~/.local/share/AppTracker/`

Там находятся `app_tracker.db`, `app_tracker.log` и служебные файлы сторожа.

## Заметки по безопасности

- Пароль на выход защищает **только выход через интерфейс**, а не завершение
  процесса через системные средства. Для последнего предназначен сторож.
- Сторож перезапускает приложение при неожиданном завершении, но **всегда**
  уважает корректный выход (через меню/пароль) и прекращает попытки при
  слишком частых перезапусках, чтобы не зациклиться.
- Хеширование пароля использует `bcrypt`, а при его отсутствии —
  salted SHA-256.
